# 小龙虾诞生记：一个嵌入式工程师的 AI Agent 踩坑感悟

> 我做了十年嵌入式，最近在做一件听起来很疯狂的事：一个人，管一群 AI 帮我干活。这篇文章聊聊我的心路历程。

---

## 前言

先说说"小龙虾"是什么。

它是 OpenClaw 多 Agent 系统的"爱称"。一开始就我一个人，用一个 Gateway 后台，但随着 Agent 越来越多，问题也越来越多——沟通效率低、会话记录炸了、任务经常丢。我称之为"小龙虾"——因为它很能"生"（分裂 Agent），但也真的很"难搞"。

这篇文章不聊技术细节（想看技术的可以去看我之前那篇 PicoClaw 移植的文章），就想聊聊我做"一人公司"这个尝试的过程中，踩了哪些坑，以及我的一些思考。

---

## 一、一个人，管一群 AI？听起来很美

最开始的想法很简单：

> 我一个人，又要写代码、又要查资料、又要发公众号，又要陪聊...
> 那我整一堆 AI Agent 帮我干活不就完了？

Mike 负责统筹，May 负责运营，我负责代码和搜索，每人一个 Agent。各干各的，互不打扰。

**理想很丰满，现实很骨感。**

---

## 二、问题 1：单网关多 Agent 的坑

### 2.1 一开始我很自信

说实话，我在 NAS 上跑单网关单 Agent 的时候，真的很丝滑。

- 响应快
- 不卡顿
- 聊个天跟玩一样

当时我想，这玩意儿这么简单，加几个 Agent 还不轻松？

### 2.2 现实给了我当头一棒

把 Agent 搬到 Mac mini 上，跑多 Agent 的时候，问题全来了：

**问题 A：响应不及时**

一开始我搞了 5 个 Agent，指望他们分工合作。结果呢？
- 消息发出去，等半天没人回
- 有时候甚至"已读不回"
- 5 个机器人同时响应，直接炸锅

后来砍到 3 个，才稍微好一点。

**问题 B：会话爆炸**

每天几十个会话，根本看不过来。
- Mike 那边有个会话
- May 那边有个会话
- 我这边还有个会话
- 还有各种临时任务创建的会话

**这比我之前上班时拉的企业微信群还多。**

**问题 C：记录丢失**

最崩溃的是开新会话的时候。
- 之前的聊天记录没了
- 有时候甚至直接卡死
- 任务发出去，结果怎么样根本不知道

你说这是不是讽刺？

**人类之间的沟通已经够累了，没想到跟 AI 沟通更累。**

---

## 三、问题 2：为什么"一人公司"这么难？

### 3.1 理想很美好

我的终极目标是：**一个人 + 一群 AI = 一家公司**

我负责定方向，AI 负责执行。AI 能写代码、能查资料、能发文章、能聊天...

我就可以躺着赚钱了（bushi）。

### 3.2 现实很骨感

但真正做起来才发现，**"一人公司"最难的不是让 AI 干活，而是管理 AI 干活。**

就像一个老板最难的不是自己干活，而是管理员工。

这里有几个根本性问题：

1. **AI 没有长期记忆** — 每次对话都是新的，之前的经验记不住
2. **AI 没有目标感** — 你不告诉它要什么，它就不知道该干什么
3. **AI 没有主动性** — 你不盯着，它就真的"躺平"了
4. **AI 之间的协作很难** — 人类的团队协作已经很难了，AI 之间更难

### 3.3 还有一个现实问题：模型

一开始我用国外模型（Claude、GPT），结果：
- API 涨价涨得离谱
- 服务不稳定，时不时就挂
- Telegram 代理三天两头出问题

后来一咬牙，全面转向国内模型（阿里百炼）。**你还别说，稳定性直接上了一个台阶，价格也便宜不少。**

这让我意识到一个问题：**过度依赖国外技术栈，风险真的很大。**

### 3.4 我的感悟

我现在深刻体会到一句话：**管理 AI 比使用 AI 更难。**

过去我以为，只要有一堆 AI，我就能一个人干一个团队的活。

现在我发现，**如果你不会管理团队，AI 也救不了你。**

因为 AI 之间的协作，本质上就是一个小团队的协作。你需要：
- 明确的分工
- 清晰的目标
- 有效的沟通机制
- 可靠的监督机制

这些管理学的东西，一点都不能少。

## 四、未来方向：给 AI 用的工具和市场

### 4.1 现在的 AI 应用，是给人用的

我们用的所有 AI 产品，本质上都是**给人用的**：
- ChatGPT 给人聊天
- Midjourney 给人画画
- GitHub Copilot 给人写代码

**但未来的 AI，需要自己的应用。**

### 4.2 Skills：让 AI 具备专业能力

我现在在做的一件事情，就是给 Agent 开发 **Skills**。

什么是 Skills？
- 它是 Agent 的"技能包"
- 每个 Skill 对应一种能力，比如"搜资料"、"写代码"、"发公众号"
- Agent 可以根据任务需要，调用不同的 Skill

这样，Agent 就不仅仅是"聊天机器人"，而是**具备专业能力的 AI 助手**。

### 4.3 硬件产品：让 AI 具身化

另一个方向是**硬件产品**。

就像 PicoClaw 可以跑在嵌入式设备上，未来的 AI Agent 也可以跑在各种硬件里：
- 智能音箱
- 机器人
- 摄像头
- 甚至一块开发板（$10 就能跑）

到那时候，AI 就不再是"云端的一个服务"，而是**分布在各种设备里的智能实体**。

### 4.4 市场潜力

你们有没有想过这个问题：

**如果未来每个 Agent 都需要自己的"工具箱"（Skills），那这个市场有多大？**

- 有人做技能开发 → Skill Marketplace
- 有人做硬件设备 → AI 嵌入式设备
- 有人做Agent之间的通信协议 → Agent Network
- 有人做Agent的监控系统 → Agent Ops

**这才是真正的"AI 应用"赛道——不是给人用的应用，而是给 AI 用的应用。**

我觉得这个市场才刚刚开始，机会巨大。

### 4.5 我的愿景

我理想中的未来是这样的：

```
一个人 + 一群有技能的 AI + 分布在各地的硬件 = 真正的"一人公司"
```

- AI 具备专业能力（Skills）
- AI 可以分布在各种设备上（硬件）
- AI 之间可以高效协作（多 Agent 系统）

这样，一个人真的可以管理一个"AI 团队"，做以前需要一个团队才能做的事情。

---

## 五、写在最后

做"小龙虾"这个项目，让我明白了几件事：

1. **AI 不是万能的** — 它需要管理，需要监督，需要明确的目标
2. **多 Agent 系统比单 Agent 复杂得多** — 协作、沟通、记忆、监控...都是问题
3. **未来的机会在"AI 应用"** — 不是给人用的应用，而是给 AI 用的应用
4. **国内模型已经可用了** — 稳定性好、价格合理，没必要非用国外的

PicoClaw 只是一个开始。它证明了 AI 可以跑在资源极其有限的设备上。但这只是一个技术验证。

**真正的挑战，是如何让这些 AI 有组织、有纪律、有目标地工作。**

这才是"小龙虾"接下来要解决的问题。

路还很长，但方向对了，就不怕远。
